## 大改第二章的引入，明确任务和剧本系统的定位

虽然我本以为自己已经搞好了，但是俗话说活到老学到老……又被自己教做游戏了

简单来说，之前的游戏循环是“玩家接任务-找信息段/写代码-提交信息段/提交代码”

但是这就导致一个问题：当剧情中玩家的行为到达一个检查点时，npc 本应自动反应，而不是等待玩家提交信息段/代码

或者更直观地说，当前的问题表现是：玩家在网站上完成了注册，那么按照设定 npc 会立刻做出反应，玩家也没有把注册结果提交给控制节点的理由。但如果玩家不提交，那就没法跟踪任务进度，进而导致这个期间缺乏中心化的玩家进度标识，导致 desktopPet 的“聊聊先前”、“随便聊聊”、“当前任务”都无法配置任务引用来作为提供内容的基准参考

我的解法是通过编排，保证任何时候总有至少一个主线任务处于激活状态，而对于叙事上不该作为任务显示的，就作为隐藏任务，并在玩家获得信息段后自动完成

虽然这看着有点 wow 当年兔子地狱的既视感，但我觉得就目前的设计来看，这确实是 ROI 最高的方案。如果硬要做事件和任务分层、将任务视为特殊事件，那反而平白带来了额外复杂度

## 桌宠系统的资源维护

之前 claude 实现的版本里，是按照模块划分资源的。比如桌宠系统就是单独的 desktop_pet 目录

但我觉得最好分架构来，章节中特有、任务中特有、章节后解锁、任务后解锁这样的桌宠回应应该和 game_content 的对应资源放一个目录（独立 tres，不合入现有资源，避免强耦合），然后随机的公共数据放 game_content 的 desktop_pet 目录下，由加载器（类似 TaskManager 加载 TaskDefinitio）进行加载

这样以后加资源的时候更好管理

也就是说，使用了如下组织形式
```
game_content/
  desktop_pet/
    global_pool.tres                          ← 全局随机，无条件
  chapter_2/
    desktop_pet/
      chapter_2_active.tres                  ← 进入 ch2 后解锁
      chapter_2_unlocked.tres               ← ch2 全部任务完成后解锁（可选）
    tasks/
      task_01_sensitive_word_filter/
        desktop_pet/
          task_01_active.tres               ← task_01 进行中时
          task_01_unlocked.tres             ← task_01 完成后解锁
```

## 桌宠系统的架构变更

先前随机消息池和任务帮助是两套系统。前者是 Array[String] 的平铺文本列表，桌宠随机抽取一条说出来，玩家没有参与感；后者是 ExampleHelpRegistry（是的，之前 claude 写的示例我懒得改类型名就一直留到现在） + HelpEntry 的独立资源体系，用整数游标（help_cursors）依次消费提示文本。

寻思了下发现这个架构不太合适：随机消息是桌宠单方面发言、玩家只能看；任务帮助里的"先前任务"选项本质上也是闲聊性质的选项，和随机消息池功能重叠；两套资源类型（HelpEntry vs 文本列表）维护起来割裂。

所以把两者统一成了基于对话树的 PetChatPool / PetChatEntry / PetChatChoice 体系。每个 pool .tres 通过 is_help 标志区分"帮助池"和"闲聊池"，结构完全相同。PetChatEntry 支持 choices[] 分支和 follow_up 线性链接，玩家可以参与对话；seen_branches 字段记录已访问的分支，支持渐进解锁。启动时通过 _scan_chat_pools() 自动扫描 game_content 各章节目录，无需手动注册注册表资源。
