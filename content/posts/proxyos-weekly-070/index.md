## 摸索 codex 的交互范式

之前我是走 copilot+claude 方案的，也为此摸索出了行之有效的交互范式

- 写需求 story 和粗略技术文档，明确目标和模块划分
- claude opus 进行检查，细化为开发检查清单
- claude sonnet 对照清单逐模块实现
- 人工检查各个数据模型是否符合预期

但这个月 copilot 改了计费标准，以至于这么干的花销都快赶上雇人了。这大半个月我都在尝试各个 agent、调研与尝试 OpenRouter 上的流行模型，结果发现性价比仍然难以接受。

后来我无奈之下转到了我一直看不上的 gpt 的 codex agent，寻思就算生产目标总是歪，至少生产力在线。

结果发现确实生产力在线，而生产目标比我想象得还歪（再次提醒了我为啥当初看不上 gpt）

之前的交互范式几乎无法正常执行

- 写需求 story 和粗略技术文档，明确目标和模块划分
- ~~claude opus~~gpt 进行检查，细化为开发检查清单
  - 有意留的 mm 豆检查不出来，只对着细节抠字眼
  - 创造性方案提不出来，反而提了个集百家之短的傻逼方案
- ~~claude sonnet~~gpt 对照清单逐模块实现
  - 让它更新开发清单的时候，跟我说“当前实现确实有偏移，开发清单列了 xxx，但现在没做 xxx”——是的，我确定我选的是 gpt-5.5
  - 实现新 app 的时候不参考现有 app，直接自己硬搓，还往有注册机制的 hub 里塞硬编码
    - 说了之后假装实现了 app，结果一看完全没有继承 app 基类，自己手搓了半套注册
- ~~人工检查各个数据模型是否符合预期~~
  - 我一行都不敢漏检查，数据模型对不代表它不会给我新搓一套并行机制出来

而且虽然上一期借助 codex 进入了疯狂 agent 模式，但是 plus 的额度比我预期的不经用，在我只遇到了一次 5h 限额的情况下，半周就用掉了一周限额。

也就是说，我在上周四 20:24 开始使用，而这周一 20:04 就用光了这一周的窗口限额，需要等到这周四 20:26 才能继续用。

摸索被迫终止，没有确认合适的交互范式——说到底还是 codex 不读现有项目结构、实现糊弄、缺乏良好工程习惯，我暂时也想不出怎么通过交互范式的优化来抑制这些问题……它表现甚至不如 glm-5.2！

## 调整 App 架构

彻底从之前的前端为主架构改到了后端为主架构

最初我想的是应用不启动时不要占资源，但是随着应用间联动的增加（如消息触发任务、任务描述里的链接触发浏览器、浏览器里的元素触发信息段/需要查询任务完成状态等等），不得不为应用添加了相当于常驻后端的 manager。

但当时仍然没有彻底切换架构，以至于形成了 view 加载存档、manager 打开 view、manager 从 view 获取存档对象并进行修改的奇怪架构

这倒是一直相安无事，因为当初的设计没准备在玩家不开对应应用的时候修改对应的存档——但这显然不是合适的架构

所以这一期将其统一到后端处理存档、前端只管展示的架构了

## Codex 的定价有点难搞……Claude Code？

于是我开始寻思要不要升到 pro，结果一看好家伙，plus $20，pro $100

虽然说是 5x 限额，但 5x 也用不完啊……OpenAI 的小心思昭然若揭。

所以问题卷土重来——怎么搞性价比最高？

我注意到 claude 的基础订阅也是$20，如果其消耗速度和 codex 差不多，那么我想应该可以正好填充这剩下的半周。而且这总该比 OpenRouter 经济吧？ 

试了之后发现确实经济

- codex 和 claude code 的订阅都是$20 一个月
- 都有 5 小时窗口总量限制和 7 天窗口总量限制
- 结果 codex 我周四晚上订的，昨天晚上就耗尽了一周的量
- 而 claude sonnet 4.6 medium 今天爆肝一天，只用了 15% 的周额度
  - 需要注意，这个和 anthropic 的 7.13 前给额外 50%每周额度的限时优惠有很大关系
- 而且完全没犯原则性错误，计划讨论全都是健康且有意义的

不同于 codex 说啥干啥但贼 tm 糊弄，claude 先在项目里转一圈确认现状，主动向我确认其不是很确定应该怎么做的部分，然后结合原始需求文档细化出工作计划，之后我就可以和 revise doc 一样批注所有不符合预期的部分，几次 revise 后确认方案无误，就可以开始跑了。

而且结果基本没有高血压，我能提出的问题全都是正常代码实现中会出现的检视意见，比如“这部分的历史实现可以先行迁移了”或者“这个名字最好改下，因为以后它也可以承载商户之外的功能”；而不是 codex 那种“你 TM 这数据模型糊弄人啊”和“你看看这数据流向通吗”或者“你这 TMD App 不继承 AppBase，自己拿头又搓一套有意思吗”甚至“你给我停下，你调不起来 headless godot 测试就准备新建项目然后把文件复制过去测是几个意思”

---

然后第二天 anthropic 就把我号 ban 了，属实是正版受害者了，傻逼 anthropic 有钱不赚王八蛋

我突然意识到为什么会有买卖自动化注册的 claude 号的行为了
- 商家注册号，比较有代表性的收费就是￥155/每月，封号的话退款`Min（原价，原价-原件* 0.2-原价/30\*封号前使用天数）`
- 用户被 anthropic ban，只能用商家的自动注册号
- anthropic 封号，退给商家$20，对商家简直是超绝理财
- 因为有利可图，这么操作的商家越来越多
- 因为这种操作越来越多，claude 的服务器压力越来越大
- 因为这种操作越来越多且 claude 的服务器压力越来越大，anthropic 进一步加强封号策略，导致更多正常用户流向违规路径
- 哦豁，正反馈循环
